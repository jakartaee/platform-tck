Introduction

This test plan documents the high level Compatibility Test Suite (CTS) test plan for the Java Message Service (JMS), a component of J2EE. This includes JMS functionality as specified in the EJB 2.1 specification, the J2EE Platform Specification version 1.4, the JMS Specification version 1.1 and the Java Servlet specification v2.4. Compatibility test cases are derived from the requirements of these specifications. The CTS tests are written to ensure application portability and compatibility among JMS implementations based on this API. The reference implementation will validate that the specifications can be implemented, will provide a reference point for the development community, and will be used for developing Compatibility Tests, which leads to a brand. This ensures Sun's, specifically Java's WORA (Write Once Run Anywhere) mission.



Goals

Provide the CTS for the functionality as defined in the J2EE 1.4:

  • Message-driven Beans (MDB)

  • JMS access from J2EE components (excluding applets)

  • JMS API

  • Distributed Transactions


Non-Goals

  • Performance

  • Scalability

  • Ease of use

  • Non-required (implementation specific) areas of the specification



Types of Compatibility Tests

There are 4 types of tests that make up the Compatibility Test Suite, these include:  

  • Signature Tests - Checks that all of the required API signatures are available for an implementation, no supersets or subsets

  • API Tests - Verifies that all of the required classes and methods behave according to the specification on a method by method basis.

  • End-To-End Tests - A new type of compatibility test that was introduced with the J2EE platform. These type of tests span across N-tiers of the J2EE platform. These tests primarily focus and depend on server-side container runtime environments for proper execution, testing and validation.

  • Interoperability Tests - A new type of compatibility test that was introduced with the J2EE platform. These type of tests span across different J2EE platforms from different vendor's to test interoperability among these J2EE vendor's platforms.

    For CTS JMS 1.1, we will be doing Signature, API, and End-To-End tests for the J2EE 1.4 platform. There are new APIs introduced in JMS 1.1, for which new tests will be added.

Sample scenarios

Scenario1: A client sends a message to a Topic A serviced by a Message-driven Bean X. Bean X updates data in Database B. The receipt of the message and database update are all committed. Also, test with a rollback.

Test with Queue, commit and rollback.

Scenario2: A client invokes the enterprise Bean X. Bean X sends a message to a JMS queue A and updates data in a database B using connections that the Deployer configured to connect with a JMS provider and a database. Then X calls another enterprise Bean Y.

Bean Y updates data in database C. Commit operations on A,B and C. Also test with a rollback. Test with Topic, commit and rollback.

Test Approach

The CTS will utilize the J2EE reference implementation to develop JMS CTS tests.

The CTS will be adding to the Porting Package for JMS Administered Objects.

The creation of JMS Administered objects is implementation specific. Adding these api's to the porting package will keep the test suite implementation independent. Licensees will be required to modify these methods to comply with their implementation.

The Signature tests will not be the same as for J2EE release 1.4 as the JMS signatures have been changed.

The API tests will be implemented as service tests and run in Application Client, EJB and Web containers. Note that the EJB and Web tests will be a subset of the Application Client tests. Refer to section 6.6 of the J2EE Platform specification version 1.4 for a list of excluded methods. And section 17.3.5 of the EJB 2.1 specification concerning the use of createQueueSession and createTopicSession.
